## 2. Модель данных (сущности и связи)

### 2.1. Таблица `profiles` (пользователи)

Расширение над `auth.users` Supabase. Создаётся триггером при регистрации.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | `UUID` | PK, ссылается на `auth.users.id` |
| `full_name` | `TEXT` | ФИО менеджера |
| `phone` | `TEXT` | Телефон (опционально) |
| `initials` | `TEXT` | Инициалы (вычисляемое: первые буквы имени и фамилии) |
| `created_at` | `TIMESTAMPTZ` | Дата регистрации |

> В MVP одна роль — все зарегистрированные пользователи являются менеджерами с полным доступом. Поле `role` не нужно.

### 2.2. Таблица `clients`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | `UUID` | PK |
| `name` | `TEXT` | Название компании / ФИО частного лица |
| `contact_person` | `TEXT` | Контактное лицо |
| `phone` | `TEXT` | Телефон |
| `email` | `TEXT` | Email |
| `notes` | `TEXT` | Заметки (опционально) |
| `created_at` | `TIMESTAMPTZ` | Дата создания |
| `created_by` | `UUID` | FK → `profiles.id` |

### 2.3. Таблица `statuses`

Фиксированный набор из 7 статусов. Заполняется при инициализации БД (сид-скриптом).

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | `UUID` | PK |
| `name` | `TEXT` | Название статуса |
| `order_index` | `INTEGER` | Порядок отображения колонки (слева направо) |
| `created_at` | `TIMESTAMPTZ` | Дата создания |

**Фиксированный набор (7 статусов):**

| order_index | name |
|-------------|------|
| 0 | Черновик |
| 1 | На согласовании |
| 2 | Подтверждён |
| 3 | В работе |
| 4 | Выезд |
| 5 | Завершён |
| 6 | Отменён |

> Статусы фиксированы. API для управления статусами (добавление/удаление/переименование) не реализуется в MVP.

profiles (1) ──→ status_history (N)  [changed_by]

## 3. Жизненный цикл заказа

### 3.1. Этапы

| # | Этап | Действие | Статус |
|---|------|----------|--------|
| 1 | Предварительная коммуникация | Вне CRM: обсуждение условий, КП, согласование меню | — |
| 2 | Создание заказа | Менеджер создаёт заказ, заполняет клиента и контакты | **Черновик** |
| 3 | Отправка формы | Менеджер отправляет ссылку, клиент заполняет технические детали | **Черновик** → **На согласовании** |
| 4 | Согласование | Менеджер проверяет данные, прикрепляет меню, утверждает | **На согласовании** → **Подтверждён** |
| 5 | Исполнение | Менеджер меняет статусы по мере выполнения | **В работе** → **Выезд** → **Завершён** |
| 6 | Отмена | В любой момент, с обязательной причиной | Любой → **Отменён** |

### 3.2. Схема переходов статусов

```
Черновик ──→ На согласовании ──→ Подтверждён ──→ В работе ──→ Выезд ──→ Завершён
    │                                                           │
    └───────────────────────────────────────────────────────────┘
                              Отменён
```

**Автоматика при смене статуса:**
- Переход в **«Отменён»** → обязательное заполнение `cancel_reason`
- Любое изменение статуса → запись в `status_history`
- Drag-and-drop карточки → мгновенное optimistic-обновление + запрос к БД

### 3.3. Публичная форма для клиента

- **Генерация:** при создании заказа генерируется `UUID` → `form_token`.
- **Ссылка:** `https://домен/form/{form_token}`
- **Поля формы:** все поля из разделов «Основные параметры», «Время», «Локация», «Оборудование», «Напитки», «Прочее» (без финансов и служебных полей).
- **После отправки:** `form_submitted = true`, форма блокируется. Повторная отправка невозможна.
- **Валидация:** обязательные поля — дата (`event_date`), время начала (`start_time`), адрес (`address`), количество человек (`guest_count`). Zod на клиенте + сервере.

## 4. Дашборд (канбан-доска)

### 4.1. Структура

```
┌──────────────────────────────────────────────────────────────────────┐
│  [Поиск] [Фильтр по дате]                        [+ Создать заказ]   │
├────────┬──────────┬──────────┬──────────┬────────┬─────────┬────────┤
│Черновик│На соглас.│Подтвержд.│ В работе │ Выезд  │Завершён │Отменён │
│ 3·45k  │ 2·30k    │ 1·15k    │ 2·25k    │ 1·12k  │ 5·80k   │ 1·5k   │
├────────┼──────────┼──────────┼──────────┼────────┼─────────┼────────┤
│ карт.  │  карт.   │  карт.   │          │        │         │        │
│ карт.  │          │          │   ...    │  ...   │   ...   │  ...   │
└────────┴──────────┴──────────┴──────────┴────────┴─────────┴────────┘
```

- **7 фиксированных колонок** — по одной на каждый статус
- **Заголовок колонки:** название статуса, количество заказов, общая сумма (`total_cost`)
- **Фильтры:** поиск по названию/контакту, фильтр по дате мероприятия
- **Обновление:** polling каждые 5 секунд + optimistic update при drag-and-drop

### 4.2. Миниатюрная карточка

| Элемент | Источник данных |
|---------|-----------------|
| Название компании / ФИО | `orders.company_name` |
| Контактное лицо | `orders.contact_person` |
| Дата и время начала | `orders.event_date` + `orders.start_time` |
| Количество человек | `orders.guest_count` |
| Формат (цветной тег) | `orders.format` |
| Общая стоимость | `orders.total_cost` |
| Индикатор срочности ⚡ | Если `event_date` = сегодня или завтра |
| Инициалы менеджера | `profiles.initials` |

### 4.3. Drag-and-drop

- Нативный HTML5 Drag and Drop API (desktop), с запасным вариантом для touch-устройств
- При переносе карточки → optimistic-обновление (карточка визуально перемещается мгновенно)
- Запрос к Supabase: `UPDATE orders SET status_id = {new_status_id} WHERE id = {order_id}`
- При ошибке → откат карточки + toast-уведомление
- Выпадающий список статуса внутри карточки — резервный способ смены статуса

### 4.4. Суммирование стоимости

- Сумма пересчитывается: при загрузке, каждом polling-обновлении, **мгновенно при drag-and-drop** (optimistic update на клиенте)
- Запрос к Supabase для получения сумм по колонкам:

```sql
SELECT s.id, s.name, COUNT(o.id) as order_count, COALESCE(SUM(o.total_cost), 0) as total_sum
FROM statuses s
LEFT JOIN orders o ON o.status_id = s.id
GROUP BY s.id, s.name, s.order_index
ORDER BY s.order_index;
```

> Настройка колонок (добавление/удаление/переименование) не входит в MVP. Статусы фиксированы.

## 5. Полная карточка заказа (страница `/orders/{id}`)

**12 разделов формы:**

| # | Раздел | Поля |
|---|--------|------|
| 1 | Клиент и контакты | `company_name`, `contact_person`, `phone`, `email` |
| 2 | Основные параметры | `event_date`, `approval_date`, `format`, `theme`, `guest_count`, `service_rounds` |
| 3 | Время | `start_time`, `end_time`, `waiter_arrival`, `equipment_delivery`, `tables_ready` |
| 4 | Локация | `address`, `access_system`, `floor`, `entry_map`, `electricity` |
| 5 | Оборудование | `tables_info`, `utility_room`, `banner` |
| 6 | Уровень мероприятия | `event_level` (СГО / Бизнес / Частное) |
| 7 | Меню | Загрузка файла меню + `menu_comment` |
| 8 | Напитки | `alcohol`, `water`, `juices`, `coffee` |
| 9 | Прочее | `hot_dishes`, `dishware`, `extra_requirements` |
| 10 | Финансы | `total_cost` (ручной ввод), `payment_status` |
| 11 | Файлы | Список загруженных документов (JPEG, PNG, PDF, DOCX, XLSX), скачать/удалить |
| 12 | История | Хронология изменений статусов из `status_history` |

**Поведение:**
- Все поля редактируемы (кроме `id`, `order_number`, `created_at`)
- Кнопка «Сохранить» внизу страницы → один запрос `PATCH /api/orders/{id}`
- Изменение статуса — через выпадающий список на странице
- Загрузка файлов — через Supabase Storage, с превью списка

## 8. Аутентификация и авторизация

### 8.1. Схема

```
Клиент (SvelteKit) ──→ Supabase Auth (email/password)
                    ──→ middleware: все авторизованные = менеджеры = полный доступ
```

### 8.2. Row Level Security (Supabase PostgreSQL)

```sql
-- Все авторизованные имеют полный доступ ко всем таблицам
CREATE POLICY "Authenticated full access" ON orders
  FOR ALL TO authenticated
  USING (true);

CREATE POLICY "Authenticated full access" ON clients
  FOR ALL TO authenticated
  USING (true);

CREATE POLICY "Authenticated full access" ON files
  FOR ALL TO authenticated
  USING (true);
```

### 8.3. Защита публичной формы

- Проверка `form_token` (UUID) — без авторизации
- Проверка `form_submitted = false` — форма не заполнена ранее
- Защита от перебора токенов — rate limiting на уровне SvelteKit middleware

| 5 | Полная карточка заказа (форма, загрузка файлов, история) | 1.5 недели |
| 6 | Публичная форма для клиента | 1 неделя |
| 7 | Тестирование и исправление ошибок | 1 неделя |
| 8 | Деплой и обучение | 2 дня |

**Итого: ~5–6 недель (1–1.5 месяца)**

## 11. Стек технологий

| Компонент | Технология |
|-----------|------------|
| **Фронтенд** | SvelteKit (адаптивная вёрстка, SPA-режим) |
| **Стили** | Tailwind CSS (самописные компоненты) |
| **Бэкенд и БД** | Supabase (PostgreSQL + Auth + Storage) |
| **Запросы к БД** | `@supabase/supabase-js` (прямые запросы, без ORM) |
| **Валидация** | Zod (общие схемы на клиенте и сервере) |
| **API** | REST-эндпоинты SvelteKit (`+server.js`) |
| **Обновление данных** | Polling каждые 5 секунд + optimistic update при drag-and-drop |
| **Хостинг** | Vercel / Netlify (фронтенд), Supabase (БД и файлы) |

## 12. Объём проекта (Scope)

### Входит в MVP
- Веб-приложение с адаптивной вёрсткой (одна роль — менеджер)
- Канбан-дашборд с 7 фиксированными колонками, drag-and-drop и polling
- Полная карточка заказа (12 разделов, 28 полей исходной формы)
- Публичная форма для клиента (одноразовая, с валидацией Zod)
- Загрузка нескольких файлов (JPEG, PNG, PDF, DOCX, XLSX)
- История изменений статусов
- Аутентификация Supabase Auth (email/пароль)

### Не входит в MVP
- ❌ Роли (администратор, повар, логист) — все пользователи = менеджеры
- ❌ Модуль задач
- ❌ Уведомления (Telegram, Email, внутренние)
- ❌ Realtime-обновления через WebSocket (заменено polling)
- ❌ Настройка статусов/колонок (фиксированный набор из 7)
- ❌ Конструктор форм (набор полей фиксирован)
- ❌ Мобильное приложение (только адаптивный сайт)
- ❌ Интеграция с платёжными системами
- ❌ Полноценный складской учёт
- ❌ Экспорт в Excel/PDF
- ❌ Календарь загрузки

## 13. Критерии приёмки
