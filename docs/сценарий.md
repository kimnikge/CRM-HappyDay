# Сценарий взаимодействия пользователя с Catering CRM

> **Дата анализа:** 28.06.2026
> **Версия приложения:** MVP (SvelteKit 2.x + Svelte 5 + Supabase + Tailwind CSS 4)
> **Роль пользователя:** Менеджер (единственная, все авторизованные имеют равные права)

---

## 1. Карта возможностей (что реализовано)

### 1.1. Вход в систему

| # | Действие | Как делает | Результат |
|---|----------|-----------|-----------|
| 1 | Войти | Переходит на `/login`, вводит email и пароль, нажимает «Войти» | Редирект на дашборд `/` |
| 2 | Выйти | Нажимает «Выйти» в Header | Сброс сессии, редирект на `/login` |

> ⚠️ **Нет в UI:** регистрация нового пользователя, восстановление пароля. Это делается через Supabase Dashboard вручную.

### 1.2. Дашборд (канбан-доска) — главная страница `/`

| # | Действие | Как делает | Результат |
|---|----------|-----------|-----------|
| 3 | Видеть все заказы | Открывает `/` — 7 колонок по статусам | Карточки заказов распределены по колонкам |
| 4 | Искать заказ | Вводит текст в поле «Поиск по названию или контакту» | Мгновенная фильтрация (debounce) |
| 5 | Фильтровать по дате | Выбирает дату в календаре | Фильтрация заказов с датой мероприятия ≥ выбранной |
| 6 | Видеть срочность | Смотрит на карточку | ⚡ — если дата мероприятия сегодня или завтра |
| 7 | Видеть сумму заказов | В заголовке каждой колонки | Количество заказов + общая сумма `total_cost` |
| 8 | Открыть заказ | Кликает по карточке | Переход на `/orders/{id}` |
| 9 | Сменить статус перетаскиванием | Тянет карточку в другую колонку | Optimistic update + API-запрос (атомарный RPC) |
| 10 | Создать заказ | Нажимает «+ Создать заказ» | Переход на `/orders/new` |

> Автообновление: polling каждые 5 секунд. При возврате вкладки — мгновенный рефреш.

### 1.3. Создание заказа — `/orders/new`

| # | Действие | Как делает | Результат |
|---|----------|-----------|-----------|
| 11 | Выбрать существующего клиента | Выпадающий список клиентов | Автозаполнение контактов |
| 12 | Создать нового клиента «на лету» | Вводит название компании (если не выбрал из списка) | Клиент создаётся автоматически при создании заказа |
| 13 | Заполнить основные поля | Поля: компания, контакт, телефон, email, дата, формат, гостей | Валидация: название компании обязательно |
| 14 | Создать заказ | Нажимает кнопку создания | Заказ сохраняется со статусом «Черновик», редирект на `/orders/{id}` |

### 1.4. Карточка заказа — `/orders/{id}`

| # | Действие | Как делает | Результат |
|---|----------|-----------|-----------|
| 15 | Просмотреть все поля | Форма сгруппирована по разделам (Клиент, Основные параметры, Время, Локация, Оборудование, Уровень, Меню, Напитки, Прочее, Финансы, Файлы, История) | Все ~28 полей + системные |
| 16 | Редактировать поля | Меняет значения в полях, нажимает «Сохранить» | PATCH `/api/orders/{id}` |
| 17 | Сменить статус | Выпадающий список статусов в карточке | PATCH `/api/orders/{id}/status` (атомарный RPC + запись в историю) |
| 18 | Загрузить файл | Кнопка загрузки → выбор файла (JPEG/PNG/PDF/DOCX/XLSX, ≤10MB) | POST `/api/files`, файл в Supabase Storage |
| 19 | Скачать файл | Клик по имени файла | Прямая ссылка на Supabase Storage (публичный URL) |
| 20 | Удалить файл | Кнопка удаления у файла | DELETE `/api/files/{id}` |
| 21 | Видеть историю статусов | Секция «История» | Хронология: дата, автор, старый/новый статус, комментарий |

### 1.5. Публичная форма для клиента — `/form/{token}`

| # | Действие | Как делает | Результат |
|---|----------|-----------|-----------|
| 22 | Открыть форму | Переходит по уникальной ссылке (токен) | Видит название компании и поля для заполнения |
| 23 | Заполнить технические детали | Заполняет: дата, время, адрес, гости, формат, напитки, оборудование и пр. | Валидация: дата, время, адрес, гости — обязательны |
| 24 | Отправить форму | Нажимает «Отправить» | Атомарная отправка (RPC): сохранение данных + авто-переход «Черновик» → «На согласовании» |
| 25 | Повторная отправка | Пытается открыть заполненную форму | Ошибка «Форма уже заполнена» (status 410) |

### 1.6. Клиенты (через API)

| # | Действие | Как делает | Результат |
|---|----------|-----------|-----------|
| 26 | Просмотреть список клиентов | GET `/api/clients` | Список всех клиентов |
| 27 | Создать клиента | POST `/api/clients` | Новый клиент |
| 28 | Обновить клиента | PATCH `/api/clients/{id}` | Обновлённые данные |

> ⚠️ Клиенты используются ТОЛЬКО через выпадающий список при создании заказа. Отдельной страницы «Клиенты» нет.

## 2. Чего не может пользователь, но должен бы (gap-анализ)

### 🔴 Критичные пробелы (ломают основные сценарии)

| # | Пробел | Почему критично | Что нужно |
|---|--------|----------------|-----------|
| G1 | **Не видно ссылку на публичную форму** | `form_token` генерируется в БД, но нигде в UI не показывается. Менеджер не может скопировать ссылку и отправить клиенту — сценарий «отправка формы» сломан на ключевом шаге | Кнопка «Отправить форму клиенту» в карточке заказа → показывает ссылку `/form/{token}` с возможностью копирования |
| G2 | **Нет причины отмены** | Поле `cancel_reason` есть в БД, но в UI отсутствует. При переводе в «Отменён» причина не запрашивается | Модальное окно с полем «Причина отмены» при смене статуса на «Отменён» |
| G3 | **Нет страницы клиентов** | Менеджер не может посмотреть/отредактировать/удалить клиента отдельно от заказа. Нет поиска по клиентам | Страница `/clients` со списком, поиском, редактированием |
| G4 | **Нельзя удалить клиента** | Есть только create/update. Ошибочно созданного клиента нельзя удалить | DELETE `/api/clients/{id}` |

### 🟠 Существенные пробелы (затрудняют работу)

| # | Пробел | Почему важно | Что нужно |
|---|--------|-------------|-----------|
| G5 | **Нет кнопки «Копировать заказ»** | Часто бывают повторные заказы от одного клиента — приходится заполнять всё заново | Кнопка «Дублировать» → новый заказ с предзаполненными полями |
| G6 | **Нет фильтрации по ответственному менеджеру** | Все видят все заказы. При росте числа заказов трудно найти «свои» | Фильтр «Мои заказы» / выбор менеджера |
| G7 | **Нет сортировки на дашборде** | Всегда сортировка по `created_at DESC`. Нельзя отсортировать по дате мероприятия | Выбор сортировки: по дате создания / по дате мероприятия |
| G8 | **Нет экспорта** | Невозможно выгрузить заказы в Excel/CSV для отчётности | Кнопка «Экспорт» → CSV со всеми полями |
| G9 | **Нет пагинации** | Все заказы загружаются разом. При 500+ заказов тормозит | Пагинация или бесконечный скролл |
| G10 | **Нет поиска по номеру заказа** | Поиск только по названию компании и контакту. Нельзя найти по `order_number` | Добавить `order_number` в поисковый индекс |

### 🟡 Умеренные пробелы (улучшают опыт)

| # | Пробел | Почему важно | Что нужно |
|---|--------|-------------|-----------|
| G11 | **Стоимость не считается автоматически** | `total_cost` — ручной ввод. Менеджер должен сам считать. Ошибки | Калькулятор или автоподстановка из тарифов |
| G12 | **Нет уведомлений о заполнении формы** | Менеджер не узнает, что клиент заполнил форму, пока не зайдёт в дашборд и не увидит смену статуса | Email/Telegram-уведомление при отправке формы |
| G13 | **Нет «закреплённых» заказов** | Важные заказы не выделяются среди остальных | Флаг «Важный» / закрепление вверху колонки |
| G14 | **Нет календарного представления** | Только канбан. Нельзя увидеть заказы по датам на календаре | Режим «Календарь» с заказами по датам мероприятий |
| G15 | **Нет истории изменений полей** | Только история статусов. Не видно, кто и когда изменил адрес, гостей и т.д. | Таблица `order_audit_log` с диффами |
| G16 | **Нет загрузки меню как отдельного файла** | Поле `menu_file_path` есть в БД, но UI для загрузки файла меню не отделён от общих файлов | Выделенная секция «Меню» с загрузкой PDF/DOCX |
| G17 | **Нет дедлайнов и напоминаний** | Нельзя установить напоминание о приближающемся мероприятии | Поле «Напомнить за N дней», email/webhook |

### 🟢 Незначительные пробелы (nice-to-have)

| # | Пробел | Что нужно |
|---|--------|-----------|
| G18 | Нет тёмной темы | Переключатель светлая/тёмная тема |
| G19 | Нет мобильного PWA | `sw.js` существует, но не настроен — оффлайн-режим |
| G20 | Нет восстановления пароля в UI | Интеграция с Supabase Auth reset flow |
| G21 | Нет страницы профиля | Редактирование ФИО, телефона, смена пароля |
| G22 | Нет дашборд-аналитики | Графики: заказы по месяцам, выручка, топ-клиенты |
| G23 | Нет массовых действий | Выделить несколько заказов → сменить статус всем |
| G24 | Нет комментариев к заказу | Внутренние комментарии менеджеров (не видны клиенту) |
| G25 | Статусы не настраиваются | Нельзя переименовать/добавить/удалить колонку (осознанно для MVP) |

## 3. Типичный сценарий рабочего дня менеджера

### 3.1. Утро: обзор текущих дел

1. Менеджер открывает `https://catering-crm.vercel.app/`
2. Система проверяет сессию → если нет, редирект на `/login`
3. Менеджер входит с email/паролем
4. Открывается дашборд — 7 колонок канбана
5. Менеджер видит: ⚡ на карточках с сегодняшними/завтрашними мероприятиями
6. Видит суммы по колонкам — оценивает объём выручки в работе
7. Находит конкретный заказ через поиск (начинает вводить название компании)

### 3.2. Новый заказ (после звонка клиента)

1. Нажимает «+ Создать заказ»
2. Если клиент новый — вводит название компании, контакты (клиент создастся автоматически)
3. Если клиент существующий — выбирает из выпадающего списка
4. Заполняет: дату мероприятия, формат (банкет/фуршет/...), количество гостей
5. Нажимает «Создать» → заказ в статусе «Черновик»
6. **⚠️ G1:** Хочет отправить форму клиенту, но... кнопки нет. Ссылка `form_token` скрыта в БД.
7. _(В текущей версии — открывает Supabase Studio, находит заказ, копирует form_token, вручную составляет ссылку)_

### 3.3. Клиент заполнил форму

1. Клиент переходит по ссылке, заполняет все технические детали, отправляет
2. RPC `submit_order_form` атомарно сохраняет данные и переводит заказ в «На согласовании»
3. **⚠️ G12:** Менеджер не получает уведомления — узнает только при следующем взгляде на дашборд
4. Менеджер видит заказ в колонке «На согласовании»
5. Открывает заказ, проверяет данные
6. Загружает файл утверждённого меню
7. Меняет статус на «Подтверждён» (выпадающий список или перетаскивание)

### 3.4. Исполнение заказа

1. В день мероприятия перетаскивает заказ в «В работе»
2. Затем в «Выезд»
3. После завершения — в «Завершён»
4. Все изменения статуса пишутся в историю с датой и автором

### 3.5. Отмена заказа

1. Менеджер перетаскивает заказ в колонку «Отменён»
2. **⚠️ G2:** Система не запрашивает причину — просто меняет статус
3. `cancel_reason` остаётся пустым

## 4. Технические ограничения (видимые пользователю)

| # | Ограничение | Влияние |
|---|------------|---------|
| T1 | Все менеджеры видят все заказы | Нельзя скрыть конфиденциальные заказы |
| T2 | Нет разграничения прав | Любой менеджер может изменить/удалить любой заказ |
| T3 | Одна роль | Нельзя дать доступ «только чтение» повару или логисту |
| T4 | Без интернета не работает | Полностью онлайн, нет оффлайн-режима |
| T5 | Один язык | Интерфейс только на русском |
| T6 | Статусы фиксированы | Нельзя адаптировать под другой бизнес-процесс |

## 5. Резюме

**Приложение в текущем состоянии — рабочий MVP с базовым функционалом:**
- ✅ Канбан-доска с 7 статусами и drag-and-drop
- ✅ Создание/редактирование заказов с ~28 полями
- ✅ Публичная форма для клиента (заполнение тех. деталей)
- ✅ Загрузка файлов (JPEG, PNG, PDF, DOCX, XLSX)
- ✅ История статусов
- ✅ Поиск и фильтрация на дашборде
- ✅ Атомарные операции (смена статуса, отправка формы — через PostgreSQL RPC)

**Главные пробелы, которые делают сценарий неполным:**
1. **G1** — не видно ссылку на публичную форму (сломан ключевой сценарий «отправка клиенту»)
2. **G2** — нет поля причины отмены
3. **G3** — нет страницы управления клиентами
4. **G12** — нет уведомлений о заполнении формы
